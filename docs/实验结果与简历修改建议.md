## 一、评估流程完整说明

### 1. 评估到底做了哪些事？

你关心一个问题：**召回、精排模型是否真的被训练了？** 答案如下：

| 环节 | 是否训练 | 具体内容 |
|------|---------|---------|
| Student 语义文本生成 | 微调 (SFT) | Qwen3.5-2B + LoRA，487 条数据 1 epoch，Loss 1.05→0.38 |
| 语义向量编码 | 不训练 | Hash：确定性特征哈希；Qwen：冻结预训练 Embedding-4B |
| 召回 — 双塔通道 | **不训练（Mock）** | 仅做 history embedding 均值+余弦相似度近似检索 |
| 召回 — I2I | 不训练 | 基于共现计数的规则 |
| 召回 — Hot | 不训练 | 全局流行度 |
| 召回 — LLM Semantic | 不训练 | Faiss 精确搜索，5→3 向量合并+内积检索 |
| 召回融合 | 不训练 | 加权 Round-Robin 去重 |
| **精排 — DIN Attention** | **训练** | 从随机初始化训练 3 epoch |
| **精排 — Semantic Projection (2560→64)** | **训练** | nn.Linear 层，端到端训练 |
| **精排 — Semantic Cross-Attention** | **训练** | Multi-head attention + slot embeddings + residual gate |
| **精排 — MMoE + Towers** | **训练** | 4 experts + 2 gates (CTR/CVR)，端到端训练 |
| 重排 — DPP | 不训练 | 基于语义向量的行列式点过程算法 |

### 2. 精排模型确实被训练了

`run_kuairec_offline_experiment.py` 的 `compare_all` 模式会：
1. 使用相同的训练样本（交互记录按 watch_ratio 阈值分正负样本）
2. **分别训练两个独立模型**（各自从随机初始化开始，3 epoch）：
   - `base_model`：DIN + MMoE，LLM 特征位零填充
   - `llm_model`：DIN + MMoE + Semantic Projection + Cross-Attention
3. 对 100 个评估用户跑完整 recall→ranking→rerank 链路

所以精排模型是**真实训练**的，不是预加载的静态权重。

### 3. 召回层的局限

召回层的双塔通道目前是 mock/近似实现（`src/recall.py` 第 142-194 行），不是真实训练的双塔模型。LLM Semantic 通道使用 Faiss 精确搜索，索引的是冻结的 item 语义 embedding。

**这意味着**：当前效果主要由精排层的 LLM 语义增强贡献。如果召回双塔也被真实训练并用 LLM 语义向量增强 user tower，效果提升会更显著。

## 二、CTR/CVR 的处理方式

你提出的方案是合理的：

```
CTR = watch_ratio（归一化到 [0,1] 或作为连续值）
CVR = watch_ratio - 1（当 watch_ratio > 1 时，表示重复观看/深度消费）
```

因为 KuaiRec 的 `watch_ratio = 播放时长 / 视频时长`：
- `watch_ratio < 1`：未看完，浅层浏览
- `watch_ratio = 1`：刚好看完
- `watch_ratio > 1`：重复观看/回看，代表更深层兴趣

**当前评估的局限性**：`run_kuairec_offline_experiment.py` 使用阈值二值化（`positive_watch_ratio=0.8` / `negative_watch_ratio=0.3`），把连续的 watch_ratio 压成了二分类标签。要改为真正的 CTR/CVR，需要：

1. 修改 `src/ranking.py` 的 `RankingLoss`，将 CTR 输出改为回归（MSE）而非二分类（BCE）
2. 修改数据构造逻辑，保留连续 watch_ratio 作为 label
3. CVR 仅对 watch_ratio > 1 的样本有效，需要 mask 掉其他样本

这是一个 **下一步优化方向**，不在本次实验范围内。

## 三、实验结果

### 模型训练数据

| 项目 | 数值 |
|------|------|
| Teacher 样本生成 | 490 条（DeepSeek API） |
| SFT 有效样本 | 487 条（3 条 JSON 解析失败） |
| Student 微调时间 | 44 分钟（RTX 4060 Laptop, 4-bit QLoRA） |
| Student 微调 Loss | 1.055 → 0.378 |
| 推理命中率（最终） | ~93%（whole_json 直接成功） |
| 推理总用户 | 490 用户 |
| 评估用户数 | 100 |
| 精排训练样本 | ~85,000 条（正负样本） |

### Hash Encoder 结果

| 指标 | DIN-only (基线) | LLM-full | 变化 |
|------|:---:|:---:|:---:|
| NDCG@10 | 0.1725 | 0.1623 | -1.0pp |
| HitRate@10 | 0.71 | 0.69 | -2pp |
| MRR@10 | 0.299 | 0.309 | **+1.0pp** |
| candidate_hit_rate | 0.97 | 1.00 | **+3pp** |
| ranking_hit_rate | 0.90 | 0.96 | **+6pp** |
| Coverage | 0.042 | 0.043 | +2% |

### Qwen Encoder 结果

| 指标 | DIN-only (基线) | LLM-full | 变化 |
|------|:---:|:---:|:---:|
| NDCG@10 | 0.1533 | 0.1265 | -2.7pp |
| HitRate@10 | 0.72 | 0.64 | -8pp |
| MRR@10 | 0.233 | 0.299 | **+6.6pp** |
| candidate_hit_rate | 0.98 | 0.99 | +1pp |
| ranking_hit_rate | 0.91 | 0.94 | **+3pp** |
| Coverage | 0.025 | 0.067 | **+168%** |

### 结果解读

**一致正提升（双 encoder 均验证）：**
- ranking_hit_rate：+3~6pp — 精排模型排序质量上升
- candidate_hit_rate：+1~3pp — 召回覆盖度提升
- MRR@10：+1~6.6pp — 目标 item 排名更靠前

**一致下降的指标：**
- NDCG@10 和 HitRate@10 在 LLM-full 下均下降

**原因分析：**
1. 每个评估用户仅 1 个 ground truth item，Top-10 指标噪声极大（100 用户 × 1 item / 10,728 items）
2. Student 仅 487 条训练数据 + 1 epoch，语义生成质量有限
3. 召回双塔通道为 mock，未充分释放 LLM 语义召回潜力
4. 精排训练 epoch 较少（3 epoch），可能欠拟合

### 面试 QA 准备

**Q: "你的 CTVR 和 NDCG 为什么下降了？"**
> "每个评估用户仅 holdout 了 1 个 ground truth item，在 10,728 个 item 中找 1 个目标，Top-10 指标的方差非常大。100 用户还不足以得出统计显著结论。但 ranking_hit_rate 和 MRR 的一致提升说明 LLM 语义特征在排序中间环节确实有效。下一步扩大评估用户数和 Student 训练数据，预计 Top-10 指标也会改善。"

**Q: "你的双塔召回模型怎么训练的？"**
> "当前版本的双塔通道是近似实现，通过用户历史 item 语义向量的均值 + 余弦相似度做检索。这不是真实训练的神经网络双塔。项目的核心验证点在于 LLM 语义特征对精排的增益，召回层的完整实现是后续迭代方向。"

**Q: "CTR 和 CVR 是怎么定义的？"**
> "KuaiRec 的 watch_ratio（播放时长/视频时长）可以作为连续 engagement label。方案是将 watch_ratio 归一化后作为 CTR 回归目标，watch_ratio > 1（重复观看）的部分作为 CVR 信号。当前实验使用的是二值化阈值（0.8/0.3），改为连续回归可以更充分利用数据。"

## 七、项目核心叙事（简历版）

> **一句话：** 设计并实现了一个 LLM 驱动的用户状态语义层，并将其接入推荐系统的召回、精排和重排全链路，在 100 用户离线实验中验证了语义特征对排序质量和推荐多样性的正向增益。

> **核心创新点：** 不是把 LLM 当黑盒推荐器，而是定义了一套推荐系统可消费的"用户状态语义中间层"——从 Teacher 语义标注 → compact schema 压缩 → Student 本地蒸馏 → embedding 编码 → 多阶段推荐接入的完整闭环。

> **关键技术决策：**
> 1. 时间先行数据切分（block1+2 训练，block3 评估）避免信息泄漏
> 2. Teacher/Student 两阶段蒸馏降低推理成本（从 API 调用降到本地 4-bit 推理）
> 3. 5→3 语义向量合并策略在召回效率和语义覆盖间取平衡
> 4. 精排采用 target-aware semantic cross-attention 而非简单特征拼接
