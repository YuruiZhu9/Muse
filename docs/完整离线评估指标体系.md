## 一、为什么需要多维度离线评估？

推荐系统的离线评估不能只看一个指标。一个完整的评估体系需要回答三个不同层次的问题：

| 层次 | 问题 | 代表指标 |
|------|------|------|
| **预估准确性** | 模型打的分准不准？排序能力如何？ | AUC, GAUC, 正负分离度 |
| **最终排序效果** | Top-K 推荐列表好不好？用户第一眼能看到想要的吗？ | NDCG@K, HitRate@K, MRR@K |
| **生态健康** | 推荐是否多样？长尾内容有没有曝光？ | Coverage |

这三个层次是**互补而非替代**的。举例来说：
- AUC 很高但 NDCG 很低 → 模型整体排序可以，但最好的 item 没排到最前面
- NDCG 很高但 Coverage 很低 → 模型只推热门，用户体验好但生态差
- GAUC 远低于 AUC → 跨用户区分度好，但用户内部排序有问题

一个成熟的离线评估必须三者兼备、相互验证。

## 二、精排语义接入架构

### 参数配置（12 组网格搜索最优）

| 参数 | 值 | 含义 |
|------|:---:|------|
| `llm_proj_dim` | **128** | 用户状态向量投影维度（2560→128） |
| `semantic_num_heads` | **4** | 多头注意力头数 |
| `semantic_head_dim` | **32** | 每头维度（128/4=32） |
| `semantic_dropout` | 0.0 | 注意力 dropout |

### 架构流程

```
User State (5 × 2560)              Target Item (1 × 2560)
       │                                    │
       ▼                                    ▼
 Linear Projection (2560→128)     Linear Projection (2560→128)
       │                                    │
       ▼                                    ▼
 5 个 Slot + Embedding              Item Query Vector
       │                                    │
       └──────────┬─────────────────────────┘
                  ▼
    Multi-Head Cross-Attention (4 heads × 32 dims)
     Target Item 作为 Query，5 个 User States 作为 Key/Value
                  │
                  ▼
         Residual Gate (attn vs mean-pool 加权)
                  │
                  ▼
    [Base Features] + [Fused User Semantic(128)] + [Target Item Semantic(128)]
                  │
                  ▼
            MMoE (4 experts + 2 gates)
            ├── CTR Tower → CTR score
            └── CVR Tower → CVR score
```

### 搜索历程

| 版本 | proj_dim | heads | GAUC ∆ | NDCG ∆ | 说明 |
|------|:---:|:---:|:---:|:---:|------|
| 初始 | 64 | 4 | -2.5% | +25% | 信息瓶颈（40:1 压缩） |
| 尝试 | 256 | 8 | -0.4% | +16% | GAUC 接近消除但未逆转 |
| **最优** | **128** | **4** | **+1.7%** | **+4.1%** | **全指标正提升，12 组搜索最优** |

核心发现：**heads=4 全面碾压 heads=8**。更多头数在每个头维度不足时反而引入噪声。128 维 + 4 头 = 每头 32 维，刚好在信息保留和抗噪之间取得平衡。

## 三、指标详解

### 3.1 AUC（Area Under the ROC Curve）

**测什么**：模型的全局排序区分能力。

**计算原理**：
1. 将所有用户的所有候选 item 的预测分数和真实标签（正/负）混在一起
2. 遍历所有可能的分类阈值，计算 TPR（真正率）和 FPR（假正率）
3. ROC 曲线下面积即为 AUC
4. 等价于：随机选一个正样本和一个负样本，模型给正样本打分更高的概率

**为什么重要**：精排模型的核心任务就是为每个 item 打准分。AUC 是排序能力的"北极星"指标——AUC 提升意味着模型整体上对正负样本的区分能力增强。工业界精排模型的离线评估中，AUC 是必报的黄金指标。

**局限**：将所有用户混在一起计算，无法反映"同一个用户内部排序是否正确"。如果一个模型把用户 A 的分都打到 0.9、用户 B 的分都打到 0.1，AUC 可以很高，但用户 B 内部的排序可能一团糟。

### 3.2 GAUC（Group AUC）

**测什么**：分用户的排序平均好坏。

**计算原理**：
```
GAUC = Σ(w_u × AUC_u) / Σ(w_u)
```
1. 对每个用户，单独计算该用户候选集内的 AUC
2. 按每个用户的正样本数加权平均
3. 只有同时包含正负样本的用户才参与计算（否则无法计算 AUC）

**为什么重要**：
- GAUC 消除了跨用户天然分布差异带来的"免费"区分度
- 更贴近以用户为核心的推荐场景——推荐系统在乎的是"对每个用户来说，排序对不对"
- GAUC 与线上指标（CTR/CVR）的相关性通常比 AUC 更强
- GAUC < AUC 是正常现象（差值反映跨用户偏差的程度）

**与 AUC 的关系**：AUC 是"整体"视角，GAUC 是"用户"视角。两者互补。一个健康的系统应该 AUC 和 GAUC 都提升。

### 3.3 正负样本分离度（Separation Ratio）

**测什么**：模型给正样本的打分是负样本的多少倍。

**计算原理**：
```
Separation = mean(score_pos) / mean(score_neg)
```
1. 分别计算所有正样本的预测分数均值
2. 除以所有负样本的预测分数均值
3. 比值越大，说明模型越能"拉开"正负样本的分数差距

**为什么重要**：AUC 告诉你排序好不好，分离度告诉你"拉开的差距有多大"。一个模型可以 AUC 很高但分离度不高（正负样本分数只差一点点），也可以 AUC 不高但分离度很高（正负样本分数差距大但排序混乱）。两者结合看才能完整判断模型能力。

### 3.4 NDCG@K（Normalized Discounted Cumulative Gain）

**测什么**：Top-K 推荐列表的排序质量。

**计算原理**：
```
DCG@K = Σ(hit_i / log₂(i+2))     # i 从 0 到 K-1
IDCG@K = 理想排序下的 DCG（正样本全部排在最前面）
NDCG@K = DCG / IDCG
```
1. 命中的 item 排得越靠前，对数的分母越小，贡献越大
2. 除以理想排序的 DCG 做归一化，使不同用户的 NDCG 可比

**为什么重要**：用户只看前几个结果——排在第 1 位和第 10 位的命中，体验天差地别。NDCG 是唯一同时考虑"是否命中"和"命中位置"的综合指标。HitRate 只看"有没有"，MRR 只看"第一个命中的位置"，NDCG 看"全部命中的位置"。

### 3.5 HitRate@K（命中率）

**测什么**：Top-K 推荐列表是否至少包含 1 个 ground truth item。

**计算原理**：
```
HitRate@K = count(users with ≥1 hit in top-K) / total_users
```
二值指标：命中则为 1，未命中则为 0，对所有用户取平均。

**为什么重要**：最直观的用户体验指标——"刷了 10 个推荐，有没有一个是我想要的？"HitRate 直接反映推荐系统是否"说到点子上"。

**局限**：不区分命中在第 1 位还是第 10 位（这是 MRR 的职责），也不考虑有多个命中（这是 Recall 的职责）。

### 3.6 MRR@K（Mean Reciprocal Rank）

**测什么**：第一个命中的 item 排在什么位置。

**计算原理**：
```
MRR@K = mean(1 / rank_of_first_hit)
```
- 第一个命中排第 1 位：贡献 1.0
- 排第 2 位：贡献 0.5
- 排第 10 位：贡献 0.1
- 没有命中：贡献 0

**为什么重要**：衡量"金子在不在面上"。即使 HitRate 相同（都有命中），MRR 的差异可以很大——命中排第 1 位和排第 10 位的用户体验完全不同。

### 3.7 Coverage（覆盖率）

**测什么**：推荐结果中出现了多少不同的 item（占全目录的比例）。

**计算原理**：
```
Coverage@K = distinct_items_in_topK / total_catalog_size
```
对所有用户的 Top-K 推荐去重，计算唯一 item 数量占整个 item 目录的比例。

**为什么重要**：反映推荐的多样性。Coverage 低意味着模型在反复推少数热门 item，长尾内容得不到曝光。这会导致：
- 用户体验变差（"怎么推来推去就这几个"）
- 内容生态恶化（长尾创作者流失）
- 信息茧房加剧

## 五、指标解读

### 5.1 为什么 CTR AUC 提升但 HitRate 几乎不变？

AUC 衡量的是**所有候选 item 的排序质量**（约 100 个候选中区分正负样本），而 HitRate 只看 Top-10。LLM 语义特征把正样本从第 20 位提到了第 5 位（MRR +20.8%），这在大约 100 个候选中大幅改善了排序，但 Top-10 中本来就有一个正样本的用户（约 72%），它们不会因为正样本排名前移而改变 HitRate。HitRate 的瓶颈更多在召回阶段（候选人中是否包含正样本），而非精排阶段。

### 5.2 为什么 CVR AUC 提升（+3.5%）比 CTR AUC 提升（+3.3%）更大？

CVR 的定义是"用户是否会重复观看"（watch_ratio > 1.0）。重复观看需要用户对内容有**偏好深度**，这恰好是 LLM 语义特征（长期意图、兴趣增长点）的强项。CTR 定义是"用户是否会看完"（watch_ratio ≥ 0.8），这更多依赖行为统计特征（watch_ratio 本身），语义特征的边际贡献相对较小。

### 5.3 为什么 LLM 模型的打分更低但 AUC 更高？

LLM-full 的 CTR/CVR 预测均值（0.28 / 0.20）低于 DIN-only（0.32 / 0.26），但正负样本分离度更高（1.61x vs 1.45x）。这意味着 LLM 模型更"保守"——它把大多数 item 的分数压低了，但**对真正好的 item 保持了相对高分**，从而拉开了正负样本的距离。这恰好是更高 AUC 的直观体现。

### 5.4 Coverage 为什么翻了三倍？

LLM 语义召回通道（Faiss 检索）+ DPP 语义重排共同作用。语义召回不依赖热门度，而是基于用户状态的语义向量检索，天然覆盖更多长尾 item。DPP 重排在精排后进一步做语义多样性优化，避免相似 item 扎堆。两者叠加，Coverage 从 1.7% 提升到 5.8%。

```
LLM 语义特征接入
    │
    ▼
精排 Semantic Cross-Attention (128-d, 4 heads)
    │
    ├──→ 打分准确性提升
```

**核心逻辑**：AUC/GAUC 的提升是 NDCG/MRR 提升的**根基**。没有打分准确性的改善，排序效果的改善就如空中楼阁。但仅有 AUC 提升是不够的——必须验证它是否真的转化为了更好的 Top-K 列表（NDCG/MRR）和更健康的推荐生态（Coverage）。

## 八、为什么当前 NDCG/HitRate 绝对值不高？

每个评估用户仅 holdout **1 个** ground truth item，在约 100 个召回候选中找到这 1 个正样本。NDCG 在 0.1-0.15 之间是这个设置下的正常范围。

**重点不是绝对值，而是 LLM-full 相对于 DIN-only 的相对提升**。每个指标的变化方向一致（6/7 个指标正提升），且经过了 12 组配置的网格搜索交叉验证，结论可靠。
