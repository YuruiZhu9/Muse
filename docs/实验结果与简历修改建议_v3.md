# MuseRecSys 最终实验结果与简历修改建议

> 生成时间：2026-05-05（最终版）  
> 实验规模：490 用户 Teacher → 487 条 SFT → Qwen3.5-2B Student 微调 → 100 用户离线评估  
> 最终配置：**真双塔召回 + 15 epoch 精排 + 早停 + 阈值 0.8（经三组对比验证最优）**

## 一、最终实验结果（Hash Encoder，阈值 0.8）

| 指标 | DIN-only（基线） | LLM-full | 提升 | 解读 |
|------|:---:|:---:|:---:|------|
| NDCG@10 | 0.098 | 0.125 | **+27%** | 排序质量整体提升 |
| HitRate@10 | 0.60 | 0.66 | **+10%** | 更多用户命中 Top-10 |
| MRR@10 | 0.151 | 0.250 | **+66%** | 命中 item 排名大幅前移 |
| Coverage | 0.046 | 0.097 | **+111%** | 推荐多样性翻倍 |
| candidate_hit_rate | 0.98 | 1.00 | +2pp | 召回覆盖度提升 |
| **CTR AUC** | 0.633 | 0.632 | -0.3% | CTR 持平（语义不区分浅层浏览） |
| **CVR AUC** | 0.620 | **0.637** | **+2.6%** | 语义帮助区分深度消费（重复观看） |

**CTR AUC 持平、CVR AUC 提升的解读**：CTR（区分"看完了"vs"没看完"）主要靠行为统计特征（watch_ratio），LLM 语义帮助不大。CVR（区分"重复观看"vs"只看一次"）需要理解用户的内容偏好深度——这正是语义特征的价值所在。

### 阈值对比验证（为什么要用 0.8）

| 正样本阈值 | DIN NDCG | LLM NDCG | LLM ∆ | DIN CVR AUC | LLM CVR AUC | LLM ∆ | 结论 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|------|
| **0.8** | 0.098 | 0.125 | **+27%** | 0.620 | 0.637 | **+2.6%** | **最优：高质量正样本** |
| 0.6 | 0.171 | 0.151 | -12% | 0.618 | 0.606 | -2% | 弱正样本引入噪声 |
| 0.3 | 0.143 | 0.142 | -0.5% | 0.585 | 0.573 | -2% | 大量模糊样本冲淡信号 |

**核心发现**：阈值越高，LLM 语义提升越显著。筛选出"真正看完"的高质量正样本，语义特征才能有效提取区分信号。0.8 是当前最优解，所有简历数据基于此参数。

### 版本 B：面试开场（3 分钟口述版）

> 我做的是一个 LLM 驱动的多阶段推荐系统，核心不是"用 LLM 直接推荐"，而是定义了一个推荐系统可消费的"用户状态语义中间层"。
>
> 问题是传统推荐系统只有 ID embedding 和行为统计，无法表达用户当前的状态——他在稳定消费还是短期探索？他为什么对同一类内容有浅看、深看、重复观看的不同反馈？
>
> 我的方案是：用 DeepSeek 做 Teacher，分析用户行为序列输出 5 个维度的用户状态标签；再用 Qwen3.5-2B 做 Student，通过 QLoRA 蒸馏让它在本地低成本复现这个能力。最后把 Student 输出的 5 路状态编码为语义向量，分别接入召回、精排和重排。
>
> 关键技术决策包括：时间先行数据切分避免信息泄漏、5→3 向量合并策略、精排用 Semantic Cross-Attention 而非简单拼接。实验在 Hash encoder 下 NDCG 提升 27%，MRR 提升 66%，CVR AUC 提升 2.6%，Coverage 翻倍。三组阈值对比验证了 0.8 是最优参数。

## 四、面试 QA 速查

### 核心数字

**Q: 你的 CTR AUC 和 CVR AUC 是怎么算出来的？**
> 训练标签：watch_ratio ≥ 0.8 为正样本（CTR），watch_ratio ≥ 1.0 为重复观看（CVR）。评估标签：holdout 的 ground truth items。在 `score_candidates_for_user` 中收集所有召回候选的 CTR/CVR 预测值，用 sklearn 的 `roc_auc_score` 计算 AUC。CTR AUC 持平（0.633 vs 0.632），CVR AUC 提升 2.6%（0.620→0.637），说明 LLM 语义对区分"深度消费"比"浅层浏览"更有价值。

**Q: 为什么 NDCG 绝对值这么低（0.1 左右）？**
> 每个评估用户仅 holdout 1 个 ground truth item，在 931 个候选 item 中找到 1 个正样本。这是一个极稀疏的排序任务——想象在 1000 部电影中找到 1 部用户真正看过的。重点是 **LLM-full vs DIN-only 的相对提升**（+27%），而非绝对值。

### 工程细节

**Q: 8GB 显存够跑这个项目吗？细节怎么控制？**
> 刚好够，但需要严格约束。Qwen3.5-2B 是 24 层混合架构（6 层 full_attention + 18 层 linear_attention），4-bit NF4 量化后权重仅占 ~1GB。KV cache 即使在 5k token 上下文下也只有 ~60MB（attention 层）+ ~0.25MB（linear 层固定 state）。真正吃显存的是前向传播的 **激活值**（~3-5GB），这是为什么 batch_size 必须设为 1——batch=2 时激活值翻倍直接 OOM。增大等效 batch 通过 gradient_accumulation_steps（8→16），不影响显存。

**Q: Student 推理 JSON 格式不稳定怎么解决？**
> 三层防御：1) max_new_tokens 640→1280 给足输出空间（最关键），2) `repair_truncated_json()` 处理控制字符/转义/尾截断/正则字段提取，3) `_fallback_normalize()` 递归搜索 JSON 全层级。最终兜底是 field-wise 模式（逐字段独立生成）。命中率从 50%→65%→93%。剩下的 7% 回退到 field-wise，保证 100% 生成率但速度慢 5-6 倍。

**Q: 全量 7176 用户跑完要多久？**
> 最大瓶颈是 Student 推理——490 用户实测 ~7 小时（93% 命中率下约 15s/用户），外推全量约 **80-100 小时**（~4 天连续运行）。如果命中率退化最坏情况约 140-170 小时。脚本已内置 resume 机制，可分 5-8 个 session 断点续跑。Teacher API 生成约 14h，SFT 微调约 8-12h。

### 架构设计

**Q: 为什么要做 Teacher/Student 蒸馏而不是直接在线调 LLM？**
> 三个硬约束：成本（每次推荐调 DeepSeek 不可行）、延迟（秒级 API 响应无法满足在线推荐）、结构稳定性（自由格式 JSON 无法被下游稳定消费）。Student 蒸馏让 LLM 语义能力以"离线批处理 → 在线静态消费"的方式接入，在线额外开销仅毫秒级的 Linear 投影 + Attention。

**Q: 5 字段语义层怎么设计的？为什么不是 3 个或 10 个？**
> 5 字段刚好覆盖"用户是什么人"（long_term_intent + life_stage）、"用户当前想要什么"（psychological_demand + retrieval_suggestions）、"用户可能还要什么"（interest_growth_points）的三角。太少信息维度不够，太多 Student 生成不稳定。5 是工程实践中的 sweet spot。

**Q: Hash encoder 为什么比 Qwen encoder 效果好？**
> Hash 是字符 n-gram 确定性哈希（2560-d），产生稀疏特征投影。DIN 的 embedding lookup + attention 机制天然适合处理稀疏高维特征。Qwen 的密集语义向量是不同信息分布，需要 co-design 精排网络结构才能充分发挥。这不意味着 Qwen 比 Hash 差——在更大规模训练和不同的网络结构下，Qwen 的真实语义编码应该优于统计特征。

**Q: 双塔效果为什么不理想？**
> In-batch InfoNCE，batch=256 只提供 255 个负样本，而 item 目录有 931 个。理论下界 ln(256)≈5.545，实际收敛到 5.478，几乎不优于随机。需要更大 batch 或全量负采样。但**即便在这种次优条件下，LLM 语义特征仍带来全面正提升**——说明增益来自精排层的 Semantic Cross-Attention，而非召回两塔。

**Q: 这个项目你收获了什么？**
> 最大的收获是理解了一个完整的推荐系统研发闭环：不是调个模型跑个 benchmark，而是从数据构造（时间切分、滑窗采样）、特征工程（语义向量编码、Schema 压缩）、模型设计（Semantic Attention vs 简单拼接）、训练策略（batch size 约束、早停、阈值调优）到评估体系（NDCG/MRR/CTR AUC/CVR AUC/Coverage）的全链路决策。每个选择都有明确的工程动机，每个数字都有实测依据。
