# MuseRecSys 项目完整开发报告

> 生成时间：2026-05-05  
> 项目定位：LLM 驱动的用户状态感知多阶段推荐系统

---

## 一、项目背景与动机

### 1.1 要解决什么问题？

传统推荐系统对用户的表示主要依赖 ID embedding、稀疏画像特征和行为序列特征。这些方法在"点击/播放/完播"层面有效，但**无法显式表达用户的语义状态**——用户当前是长期偏好的稳定消费，还是短期探索？是放松娱乐还是深度学习？为什么对同类内容会有浅看、深看、重复观看等不同反馈？

### 1.2 核心思路

**用 LLM 从时间有序的用户行为中抽取可解释的"用户状态语义层"**，再把这层语义稳定蒸馏给本地 Student 模型，最终让推荐系统获得可解释、可扩展、可工程化接入的状态特征。

### 1.3 为什么不能直接在线调用大模型？

- **成本高**：每次请求都要走长 Prompt + API 调用
- **延迟不稳定**：在线推荐无法接受秒级到十秒级波动
- **结构不稳定**：自由格式 JSON 输出难以被 embedding 和下游链路稳定消费

因此采用 **Teacher/Student 两阶段思路**：Teacher 负责高质量语义标注，Student 负责低成本本地化复现，推荐链路只消费 Student 输出。

## 三、详细开发流程

### 阶段 0：数据准备

**数据源：KuaiRec 2.0**
- `big_matrix.csv`：12,530,806 条交互，7,176 用户，10,728 item，28 个观测日期
- `small_matrix.csv`：4,676,570 条，1,411 用户（高密度全观测子集，用于验证）
- `kuairec_caption_category.csv`：视频文本/话题/分类
- `user_features_raw.csv`：用户画像

**时间先行数据切分（避免信息泄漏）：**

big_matrix 天然分为三个不连续时间块：
- **Block 1**：2020-07-05 ~ 2020-07-12（8 天）→ Teacher/Student 训练池
- **Block 2**：2020-08-01 ~ 2020-08-10（10 天）→ 验证池
- **Block 3**：2020-08-27 ~ 2020-09-05（10 天）→ **最终离线评估区（完全隔离）**

核心规则：任意 `anchor_date` 的样本只使用该日期之前的历史行为，block3 的数据在 Teacher 生成时完全不可见。

**滑窗采样机制：**
- 每个 `anchor_date`（block 内的观测日）为一个窗口
- 历史行为：所有 `date < anchor_date` 的交互（最近 30 条）
- 标签行为：当天（`date == anchor_date`）观看的视频
- Block1 有约 8 个 anchor_date，每用户最多产生 8 个窗口样本
- 全量 7,174 用户 × block1+2 约 118,884 个候选窗口

### 阶段 1：Teacher 数据生成

**模型：** DeepSeek API（`deepseek-chat`）

**输入构造：**
```text
用户基本信息（性别/年龄段/城市/设备）
+ 时间有序历史观看记录（30条，含 video_id/watch_ratio/内容文本/分类/话题）
+ 当天观看视频列表
→ 约 5.8k~6.3k 字符的 context 文本
```

**输出（Legacy Schema）：**
```json
{
  "user_status_analysis": {
    "long_term_intent": {"description": "..."},
    "psychological_demand": {"core_demand": "...", "immediate_need": "..."},
    "life_stage_hypothesis": {"stage": "...", "confidence": "..."},
    "interest_growth_points": {"emerging_signals": [...], "bridge_concepts": [...]}
  },
  "retrieval_suggestions": {"explicit_queries": [...], "implicit_keywords": [...]}
}
```

**实测数据：**
- 生成速度：~8.7s/样本（含 API 调用 + 1s 节流延迟）
- 490 用户 × 1 窗口 → 490 条，耗时约 71 分钟
- 成功率：490/490（100%），3 条因 JSON 格式问题在后续 SFT 构建时被跳过

### 阶段 2：Schema 压缩

Teacher 输出为嵌套 JSON，但 Student 应输出固定格式的紧凑结构以降低复杂度。

**Compact Schema（5 字段）：**
```json
{
  "long_term_intent": "1-2句，总结长期稳定兴趣",
  "life_stage": "1句，当前生活阶段判断",
  "psychological_demand": "1-2句，核心心理需求",
  "retrieval_suggestions": "1-2句，显式+隐式检索关键词",
  "interest_growth_points": "1-2句，潜在扩展方向"
}
```

`normalize_compact_result()` 函数自动检测 schema 类型并转换：
- Legacy Teacher schema → 递归提取嵌套字段，拼接为中文短语
- Compact Student schema → 直接映射
- 未知 schema → **递归搜索 JSON 全层级提取字段**（容错）

### 阶段 3：Student 微调

**基座模型：** `Qwen/Qwen3.5-2B-Base`
**量化方案：** 4-bit NF4（bitsandbytes QLoRA）
**LoRA 配置：** r=16, alpha=32, dropout=0.05, target_modules="all-linear"
**硬件：** NVIDIA RTX 4060 Laptop GPU（8GB VRAM）
**训练配置：** batch_size=1, grad_accum=8, 1 epoch, lr=2e-4

**实测数据：**
- 487 条 SFT 数据
- 训练耗时：44 分钟
- Loss：1.055 → 0.378
- Token 准确率：90.96%

### 阶段 4：Student 本地推理

**推理策略（两阶段 fallback）：**
1. **Whole-JSON 模式**：一次生成完整 5 字段 JSON（max_new_tokens=1280）
2. **Field-wise 模式**（兜底）：逐字段独立生成（max_new_tokens=160 × 5）

**JSON 容错机制：**
- `parse_json_content()`：标准 JSON 解析 + raw decode + 截取 brace 范围
- `repair_truncated_json()`：移除控制字符、修复转义引号、正则字段提取兜底
- `_fallback_normalize()`：递归搜索 JSON 任意层级提取 5 字段

**实测数据：**
- 490 用户推理，耗时约 7 小时（含多次中断+resume）
- Whole-JSON 命中率：50% → 65% → **93%**（经过三轮优化）
- 命中率提升关键：max_new_tokens 640→1280 + JSON 修复机制

### 阶段 5：推荐 pipeline 接入

**召回层（4 路并行）：**

| 通道 | 方法 | 训练？ | LLM 语义？ |
|------|------|:---:|:---:|
| Two-Tower | User Tower + Item Tower（MLP），InfoNCE 训练 | 训练 | 用户状态均值→User Tower 输入 |
| Item2Item | 共现计数 | 不训练 | 否 |
| Hot | 全局流行度 | 不训练 | 否 |
| LLM Semantic | Faiss IndexFlatIP，5→3 向量合并检索 | 不训练 | 是 |

**5→3 向量合并：**
- V_profile = mean(long_term_intent, life_stage) → 用户画像查询
- V_intent = mean(psychological_demand, retrieval_suggestions) → 意图查询
- V_explore = interest_growth_points → 探索查询

**融合策略：** 加权 Round-Robin（two_tower:1.0, i2i:1.0, hot:0.8, llm:1.2），最终保留 100 个候选。

**精排层：**
```
DIN Attention (target-aware 历史加权)
    ↓
[Base Features: seq_embed + target_embed + user_feat + item_feat]
    ↓                           ↓ (if LLM enabled)
[Zero Padding]              Semantic Cross-Attention
                              (2560→64 projection
                               + Multi-Head Attention
                                target item attends to 5 user states
                               + residual gate)
    ↓                           ↓
        Fused Features → MMoE (4 experts + 2 gates)
            ↓
        CTR Tower + CVR Tower
```

**重排层：** DPP（行列式点过程），λ=0.5 平衡相关性和语义多样性。

### 阶段 6：离线评估

**评估方法：**
1. 从 490 个有 Student 推理结果的用户中，按 anchor_date 切分选取 100 个评估用户
2. 每个用户保留 anchor_date 之后的交互作为 ground truth
3. 训练两个精排模型（Base: DIN-only / LLM: DIN+Semantic）各 15 epoch + 早停
4. 训练两塔召回模型（10 epoch，InfoNCE）
5. 分别跑 din_only / llm_recall_only / llm_full 三组实验
6. 对比 NDCG@10、HitRate@10、MRR@10 等指标

## 四、实验配置演变

| 版本 | 两塔 | 精排 Epoch | CTR/CVR | 早停 | Hash LLM-full vs DIN |
|------|:---:|:---:|:---:|:---:|------|
| V1（初始） | Mock（随机） | 3 | 二分类 (BCE) | 无 | NDCG: -1pp, HitRate: -2pp |
| V2（+Faiss） | Mock（centroid） | 3 | 二分类 (BCE) | 无 | NDCG: -1pp, MRR: +1pp |
| V3（真两塔） | 真训练两塔 | 15 | 二分类 (BCE) | 有 | **NDCG: +27%, MRR: +66%** |
| V4（回归） | 真训练两塔 | 15 | 连续值 (MSE) | 有 | 实验中 |

### CTR/CVR 标签演变

**V1-V3（二分类）：**
- CTR=1.0 if watch_ratio ≥ 0.8, CTR=0.0 if watch_ratio ≤ 0.3
- CVR=1.0 if watch_ratio ≥ 1.0 (repeat watch), CVR=0.0 otherwise
- Loss: BCE (binary cross-entropy)

**V4（回归）：**
- CTR = min(watch_ratio, 3.0) — watch_ratio 作为连续 engagement 信号
- CVR = max(0, min(watch_ratio-1, 2.0)) — 重复观看深度
- Loss: MSE for CTR, BCE for CVR

## 五、工程实践要点

### 5.1 硬件环境

| 项目 | 规格 |
|------|------|
| GPU | NVIDIA GeForce RTX 4060 Laptop GPU |
| 显存 | 8188 MiB（约 8GB） |
| CUDA | 13.1 |
| PyTorch | 2.10.0+cu126 |
| Python | 3.10 |

**关键限制**：8GB 显存是整个项目的硬约束。所有模型加载、微调、推理都在这个限制下设计。

### 5.2 Qwen3.5-2B 架构参数与 KV Cache

**模型架构（从 `config.json` 实测）：**
```
num_hidden_layers: 24
  - full_attention 层: 6（每 4 层中的第 4 层: 3, 7, 11, 15, 19, 23）
  - linear_attention 层: 18（Mamba 风格 SSM，其余层）
hidden_size: 2048
num_attention_heads: 8
num_key_value_heads: 2（GQA, Grouped Query Attention）
head_dim: 256
linear_key_head_dim: 128, linear_num_key_heads: 16
linear_value_head_dim: 128, linear_num_value_heads: 16
linear_conv_kernel_dim: 4
dtype: bfloat16（2 bytes/element）
```

**KV Cache 计算（逐 token 增长）：**

| 层级 | 每 token 每层 KV 元素数 | 每 token 每层字节数 (bf16) | 计算过程 |
|------|:---:|:---:|------|
| full_attention (6层) | 1024 | **2 KB** | `2 × num_kv_heads × head_dim = 2 × 2 × 256` |
| linear_attention (18层) | — | **≈14 KB（固定）** | 不随序列增长，维持固定大小递归状态 |

**不同上下文长度下全量 KV Cache / State 占用：**

| 上下文长度 | full_attention KV | linear_attention State | 合计 | 占 8GB 比例 |
|:---:|:---:|:---:|:---:|:---:|
| 1k tokens | ~12 MB | ~0.25 MB | **~12 MB** | 0.2% |
| 5k tokens | ~60 MB | ~0.25 MB | **~60 MB** | 0.7% |
| 10k tokens | ~120 MB | ~0.25 MB | **~120 MB** | 1.5% |
| 理论最大 (262k) | ~3.1 GB | ~0.25 MB | **~3.1 GB** | 39% |

**关键洞察**：KV Cache 本身**不是** 8GB 显存的主要消耗者。8GB 显存的实际去向：

| 显存组件 | 典型占用 | 说明 |
|------|:---:|------|
| 模型权重（4-bit NF4） | ~1.0 GB | 2B 参数 × 0.5 bytes（4-bit + 量化开销） |
| LoRA adapter 权重 | ~0.05 GB | r=16 的低秩矩阵，极小 |
| **激活值 / 中间缓冲区** | **~3-5 GB** | 前向传播中的 bf16 中间 tensor，随序列长度变化 |
| KV Cache / State | ~0.06-0.12 GB | 上述计算 |
| CUDA context | ~0.5-1 GB | cuBLAS/cuDNN 运行时开销 |
| 碎片与预分配 | ~0.5-1 GB | PyTorch CUDA allocator |

**长序列时显存暴涨的根因是激活值（activations），不是 KV cache。** 这也是为什么 `batch_size=1` 是最关键的显存约束——batch=2 时激活值翻倍，直接 OOM。

### 5.3 各阶段显存消耗实测（含 KV Cache 分解）

| 阶段 | 总显存 | 模型权重 | 激活值 | KV/State | 配置 |
|------|:---:|:---:|:---:|:---:|------|
| SFT 微调（5k tokens） | 6.5-7.5 GB | ~1.0 GB (4-bit) | ~4-5 GB | ~0.06 GB | batch=1, 需存梯度 |
| SFT 微调（9k tokens） | 7.5-8+ GB | ~1.0 GB | ~5-6 GB | ~0.11 GB | 接近 OOM 边界 |
| 推理（5k tokens） | ~5.5 GB | ~1.0 GB | ~3-4 GB | ~0.06 GB | no_grad 省激活 |
| 推理（9k tokens） | ~7 GB | ~1.0 GB | ~5 GB | ~0.11 GB | 长上下文安全运行 |
| 推理（fp16 5k） | ~7 GB | ~3.5 GB (fp16) | ~2-3 GB | ~0.06 GB | 权重翻倍，不推荐 |
| Qwen Embedding (4B) | 4-5 GB | ~2 GB (4-bit) | ~2 GB | ~0.1 GB | 编码模式，no_grad |
| 精排训练 | 1-2 GB | <0.1 GB | ~1 GB | 0 | ID embedding + MLP |
| 两塔训练 | 0.5-1 GB | <0.1 GB | ~0.5 GB | 0 | 轻量 MLP |

**实测对应验证（来自 `nvidia-smi` 监控）：**
- 无进程时基准占用：~1.5 GB（桌面应用：微信/Chrome/Edge/QQ/Word 等）
- Student 推理中：~5.2 GB（+3.7 GB = 模型 1GB + 激活 ~2.5GB + KV ~0.06GB + 开销）
- 上次 OOM 时：残留 Python 进程占用 ~3.5 GB + 新进程加载 → 超过 8GB

**显存优化策略：**
- SFT/推理统一使用 4-bit NF4 量化（权重从 3.5GB→1GB）
- `do_sample=False`（greedy decoding）减少激活值
- 每个用户推理完成后主动清理 CUDA cache：`gc.collect()` + `torch.cuda.empty_cache()` + `torch.cuda.ipc_collect()`
- 避免同时加载多个模型（Embedding 模型 + Student 模型不可共存）
- 增大等效 batch 通过 `gradient_accumulation_steps` 而非 `per_device_batch_size`
- **每次重新运行前用 `nvidia-smi` 检查并 `taskkill` 残留 Python 进程**——这是最常见的 OOM 原因

### 5.3 Batch Size 设置与依据

| 模型 | batch_size | grad_accum | 等效 batch | 设置依据 |
|------|:---:|:---:|:---:|------|
| **Student SFT** | 1 | 8 | 8 | 8GB 显存限制，5k token 长序列 activation 是瓶颈 |
| **Student 推理** | 1 | — | 1 | 单用户推理，无法批处理（每用户 context 不同） |
| **精排训练** | 128 | — | 128 | DIN attention 计算量小，128 可充分利用 GPU |
| **两塔训练** | 256 | — | 256 | 需大 batch 增加 in-batch negatives 数量 |

**Student SFT batch 设置逻辑**：
- 为什么 `batch_size=1` 而不是 2？因为单条样本平均 5k-6k tokens，最长 9.6k tokens，activation 显存与序列长度平方相关。batch=2 时 8GB 大概率 OOM。
- 想增大等效 batch → 调 `gradient_accumulation_steps`（8→16），显存不受影响。
- 数据量增加不增加显存（只影响训练时间），显存只看单条样本长度 + micro-batch size。

### 5.4 上下文长度控制

| 阶段 | 最大上下文 | 截断策略 | 来源 |
|------|:---:|------|------|
| Teacher 输入 | 30 条历史 | 最近 30 条（`MAX_HISTORY_PER_SAMPLE=30`） | `generate_teacher_from_kuairec.py` |
| Student 输入 | ~6k tokens | 同 Teacher 输出 context | 不截断，完整输入 |
| Student 输出 | 1280 tokens | `max_new_tokens=1280`（whole-json） | `run_local_qwen_inference.py` |
| 精排历史 | 30 条 | `MAX_HISTORY_LEN=30` | `run_kuairec_offline_experiment.py` |

**Student 上下文长度分布（487 条 Phase1 数据）：**
- 平均：~5.6k tokens
- P50：~5.1k tokens
- P90：~6.7k tokens
- P95：~9.0k tokens
- 最大：~9.6k tokens

**为什么不能无限制加长上下文？**
- 8GB 显存下，5k token 输入约 6.5-7.5GB，9k token 接近 8GB 上限
- Qwen3.5-2B 的 Linear Attention 虽比标准 Attention 更省显存，但长序列仍然是瓶颈
- 30 条历史在 KuaiRec 场景下已经是合理的窗口——用户日均互动约 90 条，30 条覆盖了足够的行为模式

### 5.5 JSON 格式不稳定问题的处理

这是 LLM 接入推荐系统的关键工程挑战——**模型输出不是 100% 符合 JSON 规范的**，尤其是微调不充分时。

**问题表现**（Phase1 实测）：
- 缺闭合括号（`}`）：截断导致，占失败案例的约 30%
- JSON 中段缺逗号/引号错误：占约 30%
- Schema 不匹配（缺字段、字段为对象而非字符串）：占约 40%

**三层防御体系：**

```
Layer 1: parse_json_content()
  ├── 标准 json.loads()
  ├── JSONDecoder.raw_decode()（容错解析）
  └── 截取 {} 范围再解析

Layer 2: repair_truncated_json()
  ├── 移除控制字符（\x00-\x1f）
  ├── 修复未转义引号
  ├── 补全缺失闭合括号（}、"}、"]} 等组合）
  └── 正则逐字段提取（兜底：在破碎 JSON 中按字段名提取值）

Layer 3: _fallback_normalize()
  └── 递归搜索 JSON 全层级，找到 5 个字段名的任意位置提取值
```

**效果：**
| 版本 | 命中率 | 人均模型调用次数 | 推理速度 |
|------|:---:|:---:|:---:|
| 无防御（V1） | 50% | 3.5 | 119s/用户 |
| Layer 1-2（V2） | 65% | 1.7 | 23s/用户 |
| **完整三层（V3）** | **93%** | **1.1** | **~15s/用户** |

**最终兜底**：如果三层防御全部失败，回退到 Field-wise 模式——对 5 个字段各做一次独立模型调用（max_new_tokens=160），再重新组装 JSON。这保证 100% 生成率但速度慢 5-6 倍。

### 5.6 两塔召回训练的工程权衡

**为什么 batch_size 只有 256？**

InfoNCE loss 的质量取决于 batch 内负样本数量。理想情况是 batch 尽可能大（全量最好）。但实际受限于：
- 用户状态向量：2560-d（mean of 5 fields）
- Item 语义向量：2560-d
- 256 × (2560 + 2560) × 4 bytes ≈ 5.2MB（可忽略）
- 真正限制是 GPU 上的矩阵乘法并行度——256 对 RTX 4060 已经饱和

**为什么 loss 收敛不理想？**

```
理论下限：ln(batch_size) = ln(256) ≈ 5.545
实际收敛：5.536 → 5.478（仅下降 0.058）
```

这意味着模型几乎没学到超越随机水平的区分性。原因是 255 个 in-batch negatives 对 931 items 的目录来说不够——正确预测概率 ≈ 1/931，但 model 被训练为 ≈ 1/256。

**改进方向：**
1. 增大 batch 到 1024+（需更多显存或 gradient accumulation on negatives）
2. 使用全局负采样（维护负样本队列，如 MoCo 风格）
3. 从精排模型提取 item embedding 反哺召回（端到端一致性）

**当前两塔在评估中的实际影响：**
虽然两塔效果有限，但 Hash encoder 下 LLM-full 依然在所有排序指标上正提升——说明增益主要来自**精排层的 Semantic Cross-Attention**，而非召回层。

### 关键发现

1. **LLM 语义特征有效**：Hash encoder 下 LLM-full 在**所有** Top-10 指标上一致正提升

## 八、面试 QA 准备

### 项目动机

**Q: 为什么做这个项目？**
> 推荐系统最大的挑战不是模型结构，而是对用户的理解。传统 ID embedding 能捕捉"用户喜欢看什么"，但无法表达"用户当前处于什么状态"。LLM 恰好擅长从行为序列中归纳状态——所以核心思路是：让 LLM 生成"用户状态语义层"，再把这层语义接入传统推荐 Pipeline。

### 架构设计

**Q: 为什么不直接在线调用 LLM？**
> 三个原因：成本（每次请求调用大模型不可行）、延迟（秒级响应无法接受）、结构稳定性（自由格式 JSON 无法被下游稳定消费）。所以采用了 Teacher/Student 蒸馏——Teacher 负责高质量标注，Student 负责本地低成本生成，在线链路只消费 Student 的 5 字段 compact 输出。

**Q: 5 个用户状态字段怎么设计的？**
> long_term_intent 和 life_stage 刻画"用户是什么样的人"，psychological_demand 和 retrieval_suggestions 刻画"用户当前想要什么"，interest_growth_points 刻画"用户可能还要什么"。5 字段刚好覆盖长期偏好/当前需求/探索方向的三角，又不至于多到 Student 无法稳定生成。

### 数据与评估

**Q: 怎么避免时间泄漏？**
> KuaiRec 数据天然分为三个不连续时间块。block1(7月)+block2(8月初) 用于 Teacher/Student 数据构造，block3(8月底-9月) 作为完全隔离的评估区。核心规则：任意 anchor_date 只使用之前的历史，block3 的数据在 Teacher 生成时不可见。

**Q: 为什么 NDCG 绝对值这么低？**
> 每个用户只 holdout 了 1 个 ground truth item，在 ~1000 个候选 item 中找到这 1 个目标，NDCG 天然很低。所以看的是**相对提升**（LLM-full vs DIN-only 的 +27%），不是绝对值。这就像 CTR 预估中的 AUC 提升——绝对值不直观，但相对值反映模型能力。

**Q: 两塔效果为什么不理想？**
> 使用 in-batch InfoNCE 训练，batch_size=256 只能提供 255 个负样本，对 931 items 的目录远远不够（理论需要接近全量负采样才能学到好的区分性表征）。这是当前最大瓶颈，但即便在这种次优条件下，LLM 语义特征依然带来了 +27% NDCG 的提升。

### 技术细节

**Q: Student 推理 JSON 格式不稳定的问题怎么解决的？**
> 三层防御：1) max_new_tokens 从 640→1280 给足空间（最关键），2) repair_truncated_json() 处理控制字符/转义引号/尾部截断，3) _fallback_normalize() 递归搜索 JSON 任意层级提取字段。最终命中率从 50% 提升到 93%。

**Q: Hash vs Qwen Encoder 为什么 Hash 效果更好？**
> Hash encoder 产生的字符 n-gram 稀疏投影与 DIN 的 embedding lookup + attention 机制存在天然的适配性——稀疏特征天然适合做 attention query。Qwen 的密集语义向量需要更大的 embedding 维度和不同的注意力结构才能充分发挥。这不意味着 Qwen 比 Hash 差，而是说明编码器和网络结构之间存在适配性问题，需要 co-design。

**Q: 如果重新做，会怎么改进？**
> 1) 两塔召回用大 batch 或全量负采样（最有价值的改进），2) Teacher 数据扩到 3000+ 条并增加 Student 微调 epoch，3) CTR/CVR 从一开始就用连续值回归而非二分类，4) Student 推理前预计算所有 context 的 token 编码（避免重复编码）。

### 项目价值

**Q: 这个项目的核心创新是什么？**
> 不是"把 LLM 当推荐器"，而是**定义了一个推荐系统可消费的"用户状态语义中间层"**。从 Teacher 标注→Schema 压缩→Student 蒸馏→Embedding 编码→多阶段接入，形成完整的数据-模型-评估闭环。任何推荐系统都可以在这个框架下接入 LLM 语义特征，而不需要改变原有的召回/精排架构。

**Q: 如果上线，推理成本如何？**
> Teacher 是一次性离线成本（API 调用生成训练数据）。Student 微调后在本地 4-bit 推理，单用户 ~15-50s（离线批处理）。在线链路完全不调用 LLM——只消费预处理好的语义向量，额外开销在召回层（Faiss 检索，毫秒级）和精排层（Linear 投影 + Attention，毫秒级）。这也是 Teacher/Student 架构最根本的工程价值。

**Q: 这个项目证明了你什么能力？**
> 系统工程能力——不是做一个 prompt demo，而是从数据构造、模型蒸馏、特征工程、多阶段推荐 Pipeline 接入到离线评估的完整链路。技术决策（如时间切分、5 字段 schema、5→3 向量合并、Semantic Attention 而非简单拼接）都有明确的工程动机，而不是随意选择。
