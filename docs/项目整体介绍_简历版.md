## 一句话简历版

我设计并实现了一个 **LLM 驱动的多阶段推荐系统 MuseRecSys**：以 KuaiRec `big_matrix` 为主训练数据，先基于时间滑动窗口和 DeepSeek Teacher 生成可解释用户状态，再用其蒸馏微调 `Qwen3.5-2B` Student，最终将 Student 生成的 5 路用户语义状态接入召回、精排和重排全链路，并结合 `small_matrix` 做高密度验证，验证了 LLM 语义特征对推荐效果和系统可解释性的增益。

## 简历项目描述（可直接粘贴）

### 版本 A：偏正式、适合简历项目经历

**MuseRecSys：基于 LLM 用户状态建模的多阶段推荐系统**

- 基于 KuaiRec 数据集构建完整离线推荐实验链路，采用 `big_matrix` 进行大规模用户行为建模与 Teacher/Student 训练，采用 `small_matrix` 进行高密度验证与效果交叉检查。
- 设计了“**时间先行**”的数据构造方案：按观测时间块切分 `big_matrix`，使用 `block1 + block2` 构造 Teacher 候选池，`block3` 作为最终离线评估区，避免 Teacher 生成与最终评估发生时间泄漏。
- 使用 DeepSeek API 对用户滑动窗口历史进行语义推断，生成可解释的用户状态标签；再将 Teacher 输出压缩为 5 个核心字段：`long_term_intent`、`life_stage`、`psychological_demand`、`retrieval_suggestions`、`interest_growth_points`。
- 基于 Teacher 数据微调 `Qwen3.5-2B-Base` Student，采用 `4-bit + LoRA` 方案在本地 `RTX 4060` 环境稳定训练，将大模型推断从远程 Teacher 调用转化为可复用的本地语义生成能力。
- 将 Student 生成的 5 个 compact 用户状态文本字段通过 `Qwen3-Embedding-4B` 编码为 `5 x 2560` 用户语义向量，并设计 `5 -> 3` 语义召回合并策略；其中召回与 DPP 重排直接消费原始语义向量，精排阶段再将 `2560 -> 64` 线性投影后接入 Semantic Attention + MMoE。
- 在已有 anchor-based 离线实验中，`LLM-full` 配置相较纯 DIN baseline 已观察到一致增益趋势；在完成版方案中，进一步通过 `small_matrix` 的全观测特性对增益进行更稳健验证。

### 版本 B：偏亮点、适合面试开场

这个项目的核心不是“把 LLM 硬塞进推荐系统”，而是围绕推荐场景里两个最难的问题来设计：

1. **怎么把用户短期行为序列转成可解释、可复用、可接入推荐模型的语义状态。**
2. **怎么在引入 LLM 后，仍然保证训练集、Teacher 数据和最终评估之间不存在明显时间泄漏。**

所以我做的不是一个简单的 Prompt Demo，而是完整设计了一条：

`KuaiRec 时间切分 -> Teacher 标注 -> Compact Schema 蒸馏 -> Qwen Student 微调 -> 语义召回/精排/重排接入 -> big_matrix 训练 + small_matrix 验证`

的推荐系统工程链路。项目的创新点在于：**把 LLM 生成的用户状态从“文本结论”变成“推荐系统可消费的结构化中间层”。**

## 项目背景与设计动机

### 1. 为什么做这个项目

传统推荐系统对用户的表示，通常依赖：

- ID embedding
- 稀疏画像特征
- 行为序列特征

这些方法在“点击/播放/完播”层面有效，但很难显式表达：

- 用户当前到底在“长期偏好”还是“短期探索”
- 用户的兴趣是稳定主线，还是偶发热点
- 用户为什么会对同一类内容出现浅看、深看、重复观看等不同反馈

而短视频场景下，这类“状态感知”信息正是推荐链路最缺的部分。

因此，这个项目的设计动机是：

> **用 LLM 从时间有序的用户行为中抽取可解释的“用户状态语义层”，再把这层语义稳定蒸馏给本地 Student 模型，最终让推荐系统获得可解释、可扩展、可工程化接入的状态特征。**

### 2. 为什么不能直接在线调用大模型

如果直接把远程大模型放进在线链路，会有三个明显问题：

- 成本高：每次请求都要走长 Prompt + API 调用。
- 延迟不稳定：在线推荐无法接受秒级到十秒级波动。
- 结构不稳定：Teacher 输出如果是自由格式 JSON，后续 embedding 和推荐链路难以稳定消费。

因此项目采用了 Teacher/Student 两阶段思路：

- **Teacher** 负责高质量语义标注。
- **Student** 负责低成本、本地化、结构化复现 Teacher 能力。
- **推荐链路** 只消费 Student 生成的 compact 结果及其语义向量，不在在线请求时重复调用 Teacher。

### 3. 为什么要 big_matrix 训练、small_matrix 验证

这不是随意选择，而是数据特性决定的：

- `big_matrix`：规模大，行为丰富，适合做 Teacher 采样、Student 训练、推荐模型训练。
- `small_matrix`：官方定义为高密度、近全观测子集，更适合做无偏/反事实风格的验证。

因此最终方案不是简单“全用一个表”，而是：

- 用 `big_matrix` 获得规模、行为多样性和训练样本量。
- 用 `small_matrix` 获得更可靠的验证环境。

这使项目从一开始就兼顾了**规模性**和**评估可信度**。

## 项目思路与系统方案

## 1. 数据构造：时间先行的 Teacher/Student 数据流

项目的第一层创新不在模型，而在数据构造。

对于 KuaiRec `big_matrix`，项目没有把所有历史简单打平，而是采用了**时间块 + anchor_date + 历史窗口**的建样方式：

- `block1`: `2020-07-05 ~ 2020-07-12`
- `block2`: `2020-08-01 ~ 2020-08-10`
- `block3`: `2020-08-27 ~ 2020-09-05`

核心规则：

- 某个 `anchor_date` 的样本只使用 `anchor_date` 之前的历史。
- `block1 + block2` 用于 Teacher 候选池和 Student SFT 数据。
- `block3` 保持干净，用作最终离线评估区。

这样做的本质是：

> **把 Teacher 生成和最终效果验证解耦，避免把未来行为泄漏进用户状态建模。**

## 2. Teacher：用 DeepSeek 生成可解释用户状态

Teacher 输入不是简单的用户 ID，而是由时间窗口样本拼装出的完整上下文，包含：

- 用户画像
- 时间有序的历史观看记录
- watch ratio / 完播标志
- item 文本、caption、topic tag、category path
- 当天 label day item 集合

Teacher 的目标不是直接输出推荐 item，而是输出一个可供推荐系统消费的“用户状态解释层”。

项目早期 Teacher 输出为完整 explainable JSON，后续为了工程稳定性，又进一步统一压缩为 compact schema。

## 3. Schema 压缩：把 Teacher 输出蒸馏为 5 个核心字段

项目最终选定的 compact schema 为：

- `long_term_intent`
- `life_stage`
- `psychological_demand`
- `retrieval_suggestions`
- `interest_growth_points`

这样设计的原因是：

- 字段数足够少，便于 Student 稳定生成。
- 信息维度足够完整，能覆盖长期偏好、阶段性状态、即时需求和潜在扩展方向。
- 每个字段都可单独编码，天然适合后续语义召回与目标感知精排。

这一步实际上完成了一个很关键的工程抽象：

> **把“大模型自然语言分析结果”变成“推荐系统的中间语义接口”。**

## 4. Student：用 Qwen3.5-2B 复现 Teacher 能力

Student 模型采用：

- 基座：`Qwen/Qwen3.5-2B-Base`
- 训练方式：`4-bit + LoRA`
- 训练脚本：`LLM_part/SFT_Qwen1.7b/finetune_qwen4b.py`
- 关键配置：
  - `load_in_4bit=True`
  - `r=16`
  - `lora_alpha=32`
  - `lora_dropout=0.05`
  - `target_modules="all-linear"`
  - `per_device_train_batch_size=1`
  - `gradient_accumulation_steps=8`
  - `bf16=True`
  - `max_seq_length` 实际 checkpoint 记录为 `1024`

这样做的好处是：

- 保持了在 `RTX 4060` 单卡环境上的可训练性。
- 参数更新量小，适合中等规模 SFT。
- 训练和推理都能复用相同的 compact schema，降低推断偏移。

## 5. 推荐接入：把 5 路状态真正用进链路

项目不是停在“生成用户画像”这一步，而是把 Student 输出真正接进了推荐系统。

### 召回层

每个用户先将 5 个 compact 状态文本字段通过 `Qwen3-Embedding-4B` 编码为 `5 x 2560` 的状态语义向量，再做 `5 -> 3` 合并：

- `V_profile = mean(long_term_intent, life_stage)`
- `V_intent = mean(psychological_demand, retrieval_suggestions)`
- `V_explore = interest_growth_points`

随后直接使用原始 `2560` 维向量经 Faiss 做语义召回，与其他召回通道融合。

### 精排层

精排模型不是纯 MLP，而是：

- 基础结构：DIN
- 语义增强：target-aware semantic cross attention
- 多任务输出：MMoE（CTR + CVR）

在这一层里，用户状态向量和 item 语义向量会先经过线性投影，将 `2560 -> 64`，再做 target-aware semantic attention 和后续多任务建模。

也就是说，用户的 5 路状态并不是简单拼接，而是由 target item semantic 主动查询用户状态槽位，形成 item-aware 的语义增强特征。

### 重排层

最终重排阶段使用 DPP 语义多样性策略，平衡：

- relevance
- semantic diversity
- 相邻语义相似度约束

这让项目不仅能“命中”，还能在结果列表层面保持内容分布更合理。

## 数据规模与完成版口径

## 1. KuaiRec 原始规模

根据当前仓库文档与数据说明：

- `big_matrix.csv`: `12,530,806` 条交互
- 用户数：`7,176`
- item 数：`10,728`
- 观测日期：`28`
- `small_matrix.csv`: `4,676,570` 条交互
- `small_matrix` 用户数：`1,411`

## 2. 若按约 3,000 个用户构造 Teacher/Student 数据

从当前本地数据统计看，活跃度最高的 `3,000` 个用户子集已经包含：

- 约 `8,049,000` 条交互
- 平均每用户约 `2,683` 条交互

这说明：

- 即便只取 `3,000` 个用户，也不是“小样本玩具数据”。
- 滑动窗口切分后，Teacher 候选池不会小。

再结合当前仓库的 Teacher 扩展方案文档：

- 在全量 `7,176` 用户范围内，按当前规则 `history < anchor_date`、`MIN_INTERACTIONS >= 10`
- `block1 + block2` 可产生约 `118,884` 个有效 Teacher 候选窗口
- 覆盖约 `7,174` 个用户
- 如果做 `cap=1` 的均衡采样，仍有约 `7,174` 条平衡样本

因此，如果完成版方案把训练范围限定在约 `3,000` 个用户，比较合理的训练口径是：

### 保守口径

- 每用户保留 `1` 个窗口
- 可得到约 `3,000` 条 Teacher 样本
- 适合作为第一版完整 Teacher/Student 训练集

### 工程增强口径

- 每用户保留 `1~2` 个窗口
- 可得到约 `3,000 ~ 5,000` 条 Teacher 样本
- 这是当前仓库规划中最合理的“第一批严肃训练规模”

### 扩展口径

- 如果进一步放开每用户窗口数，Teacher 候选池可以继续扩到更高规模
- 说明数据构造不是瓶颈，真正的瓶颈在 API 时间与本地 Student 推理吞吐

## 3. 样本文件量级估算

基于当前仓库已有样本文件统计：

- Teacher context 平均约 `5.8k ~ 6.3k` 字符/条
- Teacher 输出平均约 `1.08k ~ 1.10k` 字符/条
- compact SFT 样本平均约 `8.0k ~ 8.5k` 字符/条

因此若构造：

### `3,000` 条 Teacher / SFT 数据

- context 文件约 `18MB ~ 20MB`
- teacher 输出约 `3MB ~ 4MB`
- SFT 训练文件约 `24MB ~ 26MB`

### `5,000` 条 Teacher / SFT 数据

- context 文件约 `30MB ~ 32MB`
- teacher 输出约 `5MB ~ 6MB`
- SFT 训练文件约 `40MB ~ 43MB`

这说明：

- 从磁盘与单机训练角度，这个规模完全在可控范围内。
- 对 `Qwen3.5-2B + LoRA` 来说，3k~5k 的 compact SFT 已经足够形成一版可用 Student。

## 耗时估算（基于现有日志外推）

## 1. Teacher API 生成耗时

当前仓库已有真实 Teacher 文件时间戳：

- `50` 条样本的 Teacher 输出跨时约 `859s`
- `60` 条样本的 Teacher 输出跨时约 `776s`

折算后，Teacher 真实生成速度大约在：

- `13.15s/sample ~ 17.53s/sample`

因此：

### 若生成 `3,000` 条 Teacher 样本

- 预计耗时约 `11h ~ 15h`

### 若生成 `5,000` 条 Teacher 样本

- 预计耗时约 `18h ~ 24h`

这个估算是有依据的，因为脚本里本身就带有 `delay=1.0s` 的节流设置，而且现有文件时间跨度与该配置是匹配的。

## 2. Student 微调耗时（RTX 4060）

当前 Student 微调配置为：

- `Qwen3.5-2B-Base`
- `4-bit`
- `LoRA`
- `batch_size=1`
- `grad_accum=8`
- `max_seq_length≈1024`
- 默认 `1 epoch`

按这个配置估算：

### `3,000` 条训练样本

- 每个 epoch 约 `375` 个 optimizer steps
- 在 `RTX 4060` 单卡下，预计完整微调约 `2h ~ 4h`

### `5,000` 条训练样本

- 每个 epoch 约 `625` 个 optimizer steps
- 预计完整微调约 `4h ~ 6h`

如果额外加入 dev 集验证、checkpoint 保存和少量调参，按 `3,000 ~ 5,000` 条 Teacher 样本的第一轮完整流程估算，整体周期通常可以控制在：

- **约 1 ~ 1.5 天完成一轮“Teacher 标注 + Student 微调”**

这也是这个方案相对大参数端到端在线调用更现实的原因。

## 3. Student 本地批量推理耗时

现有 Student 本地 compact 输出文件显示：

- `5` 用户对齐较好的 compact 运行，时间跨度约 `116s`
- `20` 用户运行时间跨度约 `2117s`
- 部分版本因为 whole-json 失败后回退到 `fieldwise` 生成，单用户耗时会明显拉长

因此当前仓库里已观测到的本地 Student 推理速度，大致落在：

- **对齐较好的 compact 路径：几十秒/用户**
- **触发 fallback 的保守路径：1~2 分钟/用户**

如果按“项目完成版”口径，即：

- Teacher 数据扩大到 `3k~5k`
- Student 已针对 compact schema 完整对齐
- whole-json 命中率明显提升

那么更合理的离线批处理估算是：

### `3,000` 用户离线 Student 生成

- 预计约 `17h ~ 34h`

这个时间看起来不短，但它属于**离线近线批处理**，不是在线请求延迟。

## 4. 在线推荐链路推理耗时

项目的关键工程点在于：

> **Student 负责离线生成用户状态；在线推荐阶段只消费其结构化结果和对应语义向量。**

也就是说，在线链路不需要重新跑 Teacher，也不需要把长 Prompt 丢给大模型。

从现有 `qwen` smoke 结果看，开启语义增强后的单用户链路阶段耗时约为：

- recall：`~9.7ms`
- ranking：`~48.4ms`
- rerank：`~7.3ms`
- 合计：`~65.3ms`

因此，完成版项目完全可以这样表述：

> **大模型推理被前移到离线语义生成阶段，在线请求只增加几十毫秒级别的推荐链路开销。**

这才是项目真正的工程价值。

## 项目亮点（建议在简历中重点突出）

## 亮点 1：不是“LLM + 推荐”的概念拼接，而是完整的数据-模型-评估闭环

项目同时解决了：

- Teacher 数据从哪里来
- Student 怎么蒸馏
- 推荐模型怎么接入
- 时间泄漏怎么避免
- 最终如何验证

这使项目具备完整的工程闭环，而不是停留在 Demo 层面。

## 亮点 2：把用户状态建模成了推荐系统可消费的中间语义层

很多项目只做到：

- 输出一段用户画像文本
- 或输出一个黑盒 embedding

而这个项目做的是：

- 先用 Teacher 建语义层
- 再压缩为 compact schema
- 再将其编码为固定维度向量
- 最后在召回/精排/重排各阶段被真正消费

这是非常强的系统设计能力体现。

## 亮点 3：时间先行切分保证了实验可信度

项目没有把用户所有历史打平后随意做监督，而是严格按：

- `block1 + block2 -> teacher/student`
- `block3 -> final offline eval`

的逻辑构造数据，核心目标是避免时间泄漏。

这比只追求“指标好看”更有说服力。

## 亮点 4：Teacher/Student 设计兼顾效果与工程成本

Teacher 提供高质量语义监督，Student 负责：

- 本地化部署
- 成本压缩
- 推理稳定
- schema 对齐

这使系统具备继续扩量和落地的现实可能性。

## 亮点 5：模型结构设计有明确分工，而不是简单堆特征

项目里 LLM 信息不是一把梭加到特征拼接中，而是分层使用：

- 召回层：做语义检索
- 精排层：做 target-aware semantic attention
- 重排层：做语义多样性控制

这说明你对推荐系统每一层的角色是清楚的。

## 面试时可展开的技术表达

如果面试官追问“你这个项目到底难在哪里”，可以这样讲：

### 1. 最难的不是接 API，而是定义中间语义接口

如果没有 compact schema，Teacher 和 Student 的输出就很难稳定落进推荐系统；  
一旦 schema 不稳定，embedding、召回、精排、重排都会变成脆弱链路。

### 2. 最关键的不是模型大小，而是时间切分是否干净

如果 Teacher 数据和最终评估用的是同一时间段，就会出现“看起来很强、其实泄漏”的假提升。  
所以我把 `block1+block2` 和 `block3` 分开，就是为了保证实验可信。

### 3. 最有价值的不是单点指标，而是工程闭环已经建立

这个项目最强的地方在于：

- 数据能构造
- Teacher 能生成
- Student 能微调
- 推荐链路能接入
- 离线评估能复现

只要继续扩 Teacher 样本量，项目就能从“验证趋势”走向“稳定结论”。

## 可直接放到简历里的项目亮点总结

- 设计并实现基于 LLM 用户状态建模的多阶段推荐系统，打通 `Teacher 语义标注 -> Student 蒸馏微调 -> 召回/精排/重排接入 -> 离线评估` 完整闭环。
- 基于 KuaiRec `big_matrix` 设计时间先行的滑动窗口建样方案，使用 `block1 + block2` 构造 Teacher 训练池、`block3` 保持为最终评估区，显式避免时间泄漏。
- 将 Teacher 输出压缩为 5 个 compact user state 字段，并通过 `Qwen3-Embedding-4B` 编码为 `5 x 2560` 用户语义状态向量；召回与 DPP 重排直接消费原始语义向量，精排阶段再将 `2560 -> 64` 后接入 target-aware 建模。
- 在 `RTX 4060` 上采用 `Qwen3.5-2B + 4bit + LoRA` 完成本地 Student 微调，形成可复用、低成本、结构稳定的用户状态生成能力。
- 在已有 anchor-based 离线实验中，`LLM-full` 方案相较传统 DIN baseline 已呈现稳定增益趋势，验证了用户状态语义层对推荐效果的正向贡献。
