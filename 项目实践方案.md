# 项目实践方案

## 1. 当前结论

当前项目已经可以按 **方案 A** 运行一条真实的 student 版本最小闭环：

1. 用已有 LoRA 微调模型 `Qwen3.5-4B` 读取 KuaiRec 用户上下文  
2. 生成固定结构的 compact 用户语义 JSON  
3. 用 `Qwen3-Embedding-4B` 对语义文本和 item 文本编码  
4. 把向量送入推荐系统 smoke pipeline  

这条链路已经在仓库中真实跑通了一个最小样例。

## 3. 当前固定后的 student 输出结构

为了让 student 更稳定，当前已经把本地推理目标统一为 compact schema：

```json
{
  "user_basic_information": {
    "user_id": "0",
    "user_active_degree": "high_active",
    "gender": "F"
  },
  "long_term_intent": "...",
  "life_stage": "...",
  "psychological_demand": "...",
  "retrieval_suggestions": "...",
  "interest_growth_points": "...",
  "generated_time": "YYYY-MM-DD HH:MM:SS"
}
```

说明：

- 顶层不再要求复杂嵌套 explainable JSON
- 只保留后续 embedding 和 pipeline 真正要用的 5 个语义字段
- 这 5 个字段都固定为字符串，便于后续编码

相关实现：

- schema 定义：`LLM_part/student_schema.py`
- SFT 数据构造：`LLM_part/build_sft_dataset.py`
- 本地 student 推理：`LLM_part/run_local_qwen_inference.py`

## 6. 当前推荐系统中的语义接入方式

### 6.1 用户侧

compact JSON 中的 5 个字段会被转换成 5 个用户状态文本：

- `long_term_intent`
- `life_stage`
- `psychological_demand`
- `retrieval_suggestions`
- `interest_growth_points`

然后编码成 `shape = [num_users, 5, 2560]` 的用户状态向量。

### 6.2 item 侧

每个 item 会把以下内容拼成文本：

- 主文本
- 主类别路径
- 补充类别路径
- topic tag

然后编码成 `shape = [num_items, 2560]` 的 item 语义向量。

### 6.3 recall 层

当前 recall 会把 5 个用户状态向量合成 3 个查询向量：

- `V_profile = mean(long_term_intent, life_stage)`
- `V_intent = mean(psychological_demand, retrieval_suggestions)`
- `V_explore = interest_growth_points`

然后对 item 向量做语义召回。

## 7. 当前还没完成的部分

### 7.1 多用户 student 批量推理太慢

当前最主要的现实瓶颈不是逻辑错误，而是推理速度：

- `Qwen3.5-4B-Base + LoRA` 本地加载本身就慢
- 用户上下文很长
- 当前没有 fast path，日志显示退回到了 torch 实现
- 因此 1 用户可以完成，但 2~5 用户批量推理会明显变慢

所以现阶段建议：

- 先用 1~5 用户验证结构
- 再逐步扩量
- 不要一开始就直接跑 50 或 300 用户

### 7.2 ranking 仍然是 smoke 级别验证

当前 `run_kuairec_llm_smoke.py` 主要验证的是：

- 数据接口通不通
- recall / rank / rerank 的输入输出对不对

它不是正式效果评估脚本。

### 7.3 teacher 与 student 的完全对齐仍可继续增强

当前 student 输出已经固定到 compact schema，但如果后续要进一步提升稳定性，可以再做：

- compact schema 的二次微调
- 训练模板与推理模板完全一一对应
- 更严格的 JSON 约束生成

但这不是当前必须步骤。
